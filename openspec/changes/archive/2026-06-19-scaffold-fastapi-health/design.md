## Context

The repository currently has product, API, system design, and technology stack documents for the Signapse Market Data Gateway crypto MVP. It does not yet have a Python package, FastAPI app, project metadata, or test suite.

This change creates the first runnable slice of the gateway: an ASGI application with `GET /health`. The endpoint already exists in `docs/spec.md` and `docs/system-design.md`, and it is intentionally independent from Binance, PostgreSQL, and stream infrastructure so the service can boot before those integrations exist.

## Goals / Non-Goals

**Goals:**

- Create a minimal FastAPI application entrypoint at `app.main:app`.
- Expose `GET /health` with the documented response shape.
- Return gateway time as an ISO-8601 UTC timestamp.
- Add `uv` project metadata and baseline tool configuration.
- Add route-level test coverage for `/health`.
- Keep the scaffold compatible with the planned package layout in `docs/system-design.md`.

**Non-Goals:**

- Implement `/v1/symbols`, `/v1/quotes`, `/v1/candles`, or `/v1/stream`.
- Connect to Binance REST or WebSocket APIs.
- Connect to PostgreSQL or add Alembic migrations.
- Add Docker Compose deployment files.
- Add authentication, authorization, CORS policy, rate limiting, or public gateway hardening.
- Add a dependency-aware readiness endpoint.

## Decisions

### Use FastAPI as the first application boundary

The scaffold will create a real FastAPI app rather than a placeholder script.

Rationale:

- `docs/tech-stack.md` selects FastAPI and Uvicorn as the API/runtime stack.
- It gives the project immediate OpenAPI generation and route testing.
- Later API routes can plug into the same router structure.

Alternative considered:

- A plain ASGI callable or simple HTTP server would reduce dependencies, but it would not exercise the actual framework selected for the gateway.

### Keep `/health` as liveness only

`GET /health` will report that the process is running and include current gateway time. It will not check Binance or PostgreSQL.

Rationale:

- A first scaffold should remain runnable without external services.
- Provider/database readiness has different semantics from process liveness.
- The system design already leaves provider health as a deferred internal endpoint.

Alternative considered:

- Checking database/provider health inside `/health` would provide deeper diagnostics, but it would make the endpoint fail before those systems are implemented and would blur liveness with readiness.

### Centralize UTC time formatting

The implementation will use a small core time helper for UTC timestamps used by `/health`.

Rationale:

- The public contract requires UTC-only timestamps.
- Centralizing formatting avoids inconsistent datetime serialization as more endpoints are added.

Alternative considered:

- Returning `datetime.now(timezone.utc)` directly from the route would work initially, but it spreads contract formatting into API handlers.

### Add locked tooling and dependency versions now

The implementation should add `pyproject.toml`, `.python-version`, and tool configuration using the locked versions from `docs/tech-stack.md`. `uv.lock` should be generated by running `uv sync` during implementation/verification and committed with the scaffold.

Rationale:

- Project metadata is source code and belongs in the scaffold.
- `docs/tech-stack.md` locks Python `3.14.6`, FastAPI `0.137.2`, Pydantic `2.13.4`, pydantic-settings `2.14.1`, pytest `9.1.0`, pytest-asyncio `1.4.0`, HTTPX `0.28.1`, Ruff `0.15.18`, mypy `2.1.0`, and uv `0.11.22` as the project baseline.
- The lockfile should reflect those resolved dependencies so local development, CI, and production do not drift.

Alternative considered:

- Deferring all tooling until later would make the first runnable slice harder to verify consistently.

## Risks / Trade-offs

- Health endpoint is shallow -> keep it documented as process liveness and add a separate internal readiness/provider-health endpoint later if operations need it.
- Dependency resolution may fail if the locked Python or package versions are not available in the local environment -> pause and decide explicitly before changing versions.
- Timestamp tests can be flaky if they assert exact time -> test parseability, UTC suffix, and response shape instead of exact clock equality.
- Minimal scaffold does not prove Binance or PostgreSQL integration -> keep those as separate follow-up changes with their own tests.

## Migration Plan

1. Add project metadata and dependency configuration.
2. Add FastAPI app package and `/health` route.
3. Add tests and run verification commands.
4. Start the app locally with Uvicorn to confirm the route is served.

Rollback is simple: remove the scaffold files introduced by this change. No data migration, external service state, or API deprecation is involved.
